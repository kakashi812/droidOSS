Chapter Ten building

Finding each other

Typing 192.168.1.12 into a phone is the roughest edge in the entire product. Here's how to remove it — and how much effort that's worth.

Builds on Ch 5: IP addresses and ports. And Ch 8: the DISCOVER message type we reserved.

The problem, stated fairly

Your phone needs the server's IP address. It has no way to guess it — addresses are assigned by your router and change over time. So either the user supplies it, or the phone goes looking.

This is worth solving well because it's the very first thing anyone does with your app, and first impressions are disproportionate. But it's also worth not over-solving, because every tier of sophistication brings new ways to fail.

Tier 1 — type the IP

Genuinely: build this first, today. It takes ten minutes and it unblocks all your other testing. And keep it permanently as a fallback, because it's the thing that still works when the network is doing something strange.

Make the server display its own address prominently. Finding it is one call:

var ip = Dns.GetHostEntry(Dns.GetHostName())
            .AddressList
            .First(a => a.AddressFamily == AddressFamily.InterNetwork);
Console.WriteLine($"Connect your phone to:  {ip}:27500");

Tier 2 — broadcast

Networks have a special address meaning "everyone on this network." Send there and every device receives it. Our server is listening; nothing else cares.

Broadcast

The address 255.255.255.255 means "deliver this to every device on the local network." It only works with UDP — the whole idea is incompatible with TCP's one-to-one connections.

It cannot leave your local network. Routers refuse to forward broadcasts, which is exactly the containment we want.

// Phone: ask
val sock = DatagramSocket().apply { broadcast = true; soTimeout = 800 }
val ask  = byteArrayOf(0xDA.toByte(), PROTO_VERSION, MSG_DISCOVER, 0)
sock.send(DatagramPacket(ask, ask.size,
          InetAddress.getByName("255.255.255.255"), DISCOVERY_PORT))

// Then collect replies until the timeout expires
val servers = mutableListOf<InetAddress>()
val reply = DatagramPacket(ByteArray(64), 64)
while (true) {
    try { sock.receive(reply) } catch (e: SocketTimeoutException) { break }
    if (reply.data[0] == 0xDA.toByte() && reply.data[2] == MSG_WELCOME)
        servers += reply.address
}
When broadcast fails

Some routers enable AP isolation, which deliberately stops wireless devices from talking to each other. It's common on guest networks and in hotels and student accommodation.

Nothing you write can defeat it — it's the network refusing, by design. This is precisely why Tier 1 stays in the app forever, and why your error message should say "no servers found — try entering the address manually" rather than just failing.

Tier 3 — mDNS

The standards-based answer. Your server advertises a service name like _padserver._udp.local, and Android discovers it through NsdManager. It's how printers and Chromecasts announce themselves.

It's better in principle: no hand-rolled protocol, works across some network configurations where raw broadcast doesn't. But on Windows it needs either Apple's Bonjour service present or you bundling an mDNS library, and it's more machinery to debug.

Recommendation

Ship Tier 1 in week one. Add Tier 2 once the core works. Only reach for Tier 3 if real users report broadcast failing — and by then you'll have the diagnostic information to know whether it would actually help.

The security question

Let's be straight about this rather than either ignoring it or over-engineering it.

You have an open UDP port that drives a virtual controller. Anyone on your network can send packets to it and move your character. Is that a problem?

Threat model

An honest description of who might attack you, what they could gain, and what it would cost them. Security work without one tends to produce elaborate defences against irrelevant threats while leaving obvious gaps.

Ours: people already on your home network, whose worst achievable outcome is being irritating. That deserves a proportionate response, not encryption.

The proportionate answer is a shared secret. The server shows a QR code containing ip:port:token; the phone scans it; every packet carries the token; the server drops packets without it.

One thing not to do

Do not forward this port through your router so you can play from outside the house. Internet latency makes it unplayable anyway (Chapter 6), and you'd be exposing an unauthenticated input device to the entire internet.

This is LAN-only software by nature. That's a feature.

Should you add encryption?judgement call

Probably not, and there's an instructive precedent. The VirtualGamePad project has encryption on its roadmap, with a note that a security professional told the author it's unnecessary for this use case.

That's the right call, for three reasons:

  • Nothing secret is being transmitted. The payload is "the left stick is at -31000." Intercepting it reveals that someone is playing a game.
  • An attacker is already inside. To capture your packets they need to be on your Wi-Fi. Someone with that access has far more interesting targets than your gamepad.
  • It costs latency and complexity in the one loop where you've worked hardest to have neither.

A shared token is the right amount of protection. Knowing when to stop is a real engineering skill, and this is a clean example of it.

What you now know
  1. Discovery comes in three tiers: manual entry, UDP broadcast, and mDNS.
  2. Build manual entry first and never remove it — it's the fallback that always works.
  3. Broadcast to 255.255.255.255 reaches every device on the network, works only over UDP, and can't escape your LAN.
  4. AP isolation defeats broadcast entirely and you can't work around it — so fail with a helpful message.
  5. A threat model keeps security effort proportionate. Ours is "people already on your Wi-Fi being annoying."
  6. A QR code delivers address, port, and a shared token in one gesture — solving discovery and pairing together.
  7. Encryption is not warranted here, and knowing where to stop is part of the job.
