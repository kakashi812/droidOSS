Chapter Two gentle

How a game hears a button press

This is the most important chapter in the book. Everything else is built on the idea in it.

Builds on Ch 1: controllers report complete state, not events.

The question worth asking

Forget phones for a moment. Take an ordinary Xbox controller, plug it into a PC by USB, launch a game, and press Ⓐ. Your character jumps.

Ask the naive question: how did the game find out?

The intuitive answer is that the game is somehow watching the controller. It isn't. Not even slightly. And once you see what actually happens, the entire phone project stops looking impossible and starts looking almost easy.

Games never talk to controllers

A game does not know what a USB port is. It has no code for reading electrical signals, no idea what brand of controller you own, and no way to communicate with hardware directly — modern operating systems specifically forbid ordinary programs from doing that.

Instead, the game asks Windows. It calls a function that Windows provides, roughly meaning: "Hey — what is controller number 0 doing right now?" Windows replies with a snapshot. The game reads it and moves on.

Think of it this way

You're at a restaurant table. You want food, but you don't walk into the kitchen — you tell the waiter, and food arrives.

You have no idea who cooked it, what the stove looks like, or whether the kitchen even has a stove. You asked the waiter; you got food. That's the whole of your relationship with the kitchen.

The game is you. Windows is the waiter. The controller is the kitchen. And critically: if the waiter brings you food, you have absolutely no way to tell whether it came from a kitchen at all.

API

An API (Application Programming Interface) is just a menu of functions one piece of software offers to another. "Here are the things you may ask me to do, and here's the format of the answer."

The waiter's menu is an API. So is the set of functions Windows offers for reading controllers.

XInput

XInput is the specific Windows API for Xbox-style controllers. When a game wants controller data, it calls a function from XInput named XInputGetState, passing a controller number (0 to 3).

You'll see this name constantly from here on. It's the front door our whole project walks through.

The full chain, all the way down

So the game asks XInput. But where does XInput get its answer? There are more layers below, and we need to see them, because that's where we'll eventually break in.

Driver

A driver is a small piece of software whose only job is to let the operating system talk to one particular kind of hardware. Your printer has one. Your graphics card has one. Your controller has one.

Drivers run in a specially privileged mode called kernel mode, which is Windows' inner circle — code there can touch hardware directly and is trusted absolutely. Ordinary programs run in user mode and cannot.

Kernel mode vs user mode

The kernel is the core of the operating system. Code running in kernel mode has unrestricted power over the machine. Everything you normally run — games, browsers, your server program — runs in user mode, walled off from the hardware, and must politely ask the kernel for anything real.

The wall between them is why you can't just write a program that pretends to be a USB device. That's a kernel-mode privilege. Remember this — it shapes Chapter 3 entirely.

The gap in the game's knowledge

Now look carefully at that diagram and notice something the game cannot do.

The game calls XInputGetState(0, &state) and receives a snapshot. There is no second function it can call to ask "…and was that a genuine Microsoft controller?" No serial number check. No authenticity test. That function does not exist, because nobody ever needed it.

The realisation

If we can get a convincing answer to appear at the driver layer, then XInput will pass it up, and the game will accept it — not because we tricked the game, but because the game is doing exactly what it always does and has no mechanism to do otherwise.

We don't need to touch the game. We don't need to modify Steam. We need to become a convincing answer, low enough down the stack.

Wait, what about DirectInput? And what's HID?optional detail

You'll bump into two other names, so here they are:

DirectInput is the older Windows API, from before Xbox controllers existed. It's more general — it can describe joysticks with weird numbers of axes and thirty buttons — but it's clumsier, it doesn't handle Xbox triggers properly (it merges the two triggers into a single axis, so you can't press both at once), and modern games have largely moved on. We're targeting XInput.

HID (Human Interface Device) is the underlying USB standard for input hardware generally — keyboards, mice, joysticks, everything. It's the layer beneath both APIs. A device describes itself with a "report descriptor" saying "I have this many buttons and these axes," and the OS figures out the rest. You don't need to write one for this project, because we'll be emulating a specific known controller rather than inventing a new one.

Practical upshot: target XInput, get everything. Games that only speak DirectInput are rare and old, and Steam can translate for them anyway.

What you now know
  1. Games never communicate with controllers. They call an API that the operating system provides.
  2. On Windows, that API is XInput, and the specific call is XInputGetState.
  3. Beneath XInput sits a driver, running in privileged kernel mode, which is what actually deals with hardware.
  4. Ordinary programs run in user mode and are walled off from hardware — which is why we can't simply declare ourselves a USB device.
  5. A game has no way to ask whether a controller is real. That question isn't in the API.
  6. Therefore: produce a convincing answer at the driver layer, and every game accepts it, automatically and permanently.
