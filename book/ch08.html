Designing the message

You're sending twenty bytes, 125 times a second, forever. This chapter decides exactly which twenty.

Builds on Ch 7: UDP, full state per packet, sequence numbers. And Ch 4: the 12-byte controller state Windows expects.

What a protocol is

Protocol

An agreement between two programs about what the bytes mean. Not a technology or a library — just a shared convention. "Byte 8 and 9 are the button bitmask" is a protocol, provided both sides believe it.

Get it wrong on one side and you don't get an error message. You get a stick that moves when you press Ⓑ, and a long confusing evening.

Why not JSON?

The instinct from web programming is to send something readable:

{ "buttons": 4096, "leftStickX": -31000, "leftStickY": 0 }

It's self-describing and easy to debug. For most software it's the right default. Here it isn't, and it's worth understanding why rather than taking it on faith.

The honest reason

It is not mainly about speed. Chapter 6 established that parsing time is irrelevant at this scale — JSON would work fine.

The real argument is allocation. JSON parsing creates new objects for every packet. At 125 packets a second, that's a steady stream of garbage for the runtime to clean up, and garbage collection is one of the few things that can cause a genuine, visible latency spike. Fixed binary lets you reuse a single buffer forever and never allocate at all.

The second argument is that our schema is fixed by hardware. A gamepad has had the same buttons for twenty years. The flexibility JSON buys us is flexibility we have no use for.

The packet

The design decision worth noticing

Bytes 8–19 are laid out exactly like the XINPUT_GAMEPAD structure from Chapter 4 — same order, same sizes, same ranges. That's deliberate.

It means the server does no conversion at all. It reads bytes off the wire and hands them almost directly to the driver. Fewer lines, fewer bugs, and nowhere for a units mix-up to hide.

General principle: when you control both ends, shape your wire format to match your output format. If you add a Linux backend later, that layer does the converting — because it's the exception, not the common case.

Size, in perspective

The other messages

Input is the constant stream, but a few other things need saying occasionally. Byte 2 tells them apart.

TypeDirectionPurpose
0x01 INPUTphone → PCThe 20-byte snapshot. 125 times a second.
0x02 HELLOphone → PC"I'm here." Server assigns a pad slot and plugs in a virtual controller.
0x03 WELCOMEPC → phone"You're pad 2." Also proves to the phone that a server exists here.
0x04 BYEphone → PCClean exit — unplug now rather than waiting for a timeout.
0x05 RUMBLEPC → phoneVibration intensity from the game. Chapter 11.
0x06 DISCOVERbroadcast"Any servers out there?" Chapter 10.
Notice what isn't there

There's no "are you still alive?" message. There doesn't need to be — the input stream is the heartbeat. A packet every 8 ms means silence is unmistakable: no INPUT for two seconds and the client is gone.

This is the second time fixed-rate sending has paid for itself, and it won't be the last. Chapter 11 builds the most important robustness behaviour in the whole project on exactly this.

Reading and writing it

// C# server — reading. No allocation, no parsing.
if (buf[0] != 0xDA || buf[1] != PROTO_VERSION) return;   // not ours
if (buf[2] != MSG_INPUT) { HandleControl(buf); return; }

uint seq     = BitConverter.ToUInt32(buf, 4);
if ((int)(seq - lastSeq) <= 0) return;                   // stale — Ch 7
lastSeq = seq;

state.Buttons = BitConverter.ToUInt16(buf, 8);
state.LeftTrigger  = buf[10];
state.RightTrigger = buf[11];
state.ThumbLX = BitConverter.ToInt16(buf, 12);
state.ThumbLY = BitConverter.ToInt16(buf, 14);
state.ThumbRX = BitConverter.ToInt16(buf, 16);
state.ThumbRY = BitConverter.ToInt16(buf, 18);
Endianness — the one trap in binary protocolsimportant

A 16-bit number occupies two bytes, and there are two conventions for which order they go in. This is called endianness.

the number 1000 (0x03E8) can be stored as:
  little-endian:  E8 03      ← least significant byte first
  big-endian:     03 E8      ← most significant byte first

Write little-endian and read big-endian and your number becomes 59395 instead of 1000. No error, no crash — just a stick behaving bizarrely.

For us it's easy: ARM (Android) and x86 (Windows) are both little-endian natively, so choosing little-endian costs nothing on either side. BitConverter in C# and ByteBuffer.LITTLE_ENDIAN in Kotlin will then agree by default.

The trap is that Java and Kotlin's ByteBuffer defaults to big-endian. Forget to set it and everything silently breaks. Write it down in your protocol notes, set it explicitly on the Kotlin side, and never think about it again.

Why a magic byte and a version number are worth 2 bytesdesign rationale

The magic byte (0xDA). Your socket will receive things you didn't send — port scanners, another app's stray broadcast, a mis-typed IP address from a different program. Without a marker byte, garbage gets interpreted as a controller state and your character spasms. One byte turns "mysterious glitch" into "silently ignored."

The version byte. Someday you'll change the packet format — an extra axis, a new field. Users will have the old phone app and the new server, in every combination. With a version byte the server can say "this client is too old, please update." Without it, an old client's packets get read with the new layout and produce nonsense.

It costs two bytes out of twenty and eliminates two whole categories of confusing bug report. Take the deal.

What you now know
  1. A protocol is just an agreement about what the bytes mean — and mismatches fail silently, not loudly.
  2. Fixed binary beats JSON here, mainly because it lets you reuse one buffer and never allocate, avoiding garbage-collection spikes.
  3. The packet is 20 bytes: 4 header, 4 sequence, 12 payload.
  4. The payload is laid out to match XINPUT_GAMEPAD exactly, so the server converts nothing.
  5. Bandwidth is irrelevant — 250× less than a video stream. Optimise for clarity.
  6. Five other message types handle joining, leaving, rumble and discovery — but there's no heartbeat, because the input stream already is one.
  7. Pick little-endian and set it explicitly on the Kotlin side, which defaults the other way.
  8. A magic byte and a version byte cost almost nothing and prevent two classes of baffling bug.
