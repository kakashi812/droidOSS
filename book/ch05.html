Chapter Five gentle

Two devices, one conversation

Networking from zero. What an IP address and a port really are, what a socket does, and how two programs on different machines find each other at all.

Builds on Ch 4: you have a working virtual pad driven by fake data. Assumed knowledge: none — this chapter starts from nothing.

The problem

Your server can move a virtual stick. Your phone knows where a thumb is. These two facts are currently in two different buildings, so to speak. This chapter is about the road between them.

Addresses: finding the right machine

Every device on your home network has a number that identifies it, handed out by your router when the device connects. That's an IP address — typically something like 192.168.1.47.

IP address

A number identifying one device on a network. Addresses starting 192.168. or 10. are private — they only mean anything within your own network, and every home in the world reuses the same ranges.

That's genuinely useful for us: it means our whole system is naturally confined to your home network. It also means both devices must be on the same Wi-Fi, which will be the single most common support question you ever get.

Ports: finding the right program

An IP address gets you to the machine. But a PC runs dozens of programs at once — a browser, Discord, Steam, your server. When data arrives, which one gets it?

That's what a port is: a numbered slot on a machine that a program can claim.

Think of it this way

The IP address is the street address of an apartment building. The port is the flat number.

"Deliver to 192.168.1.12, port 27500" means the same kind of thing as "Deliver to 14 Oak Street, flat 12." Without the flat number the postman is standing in the lobby holding a parcel with nowhere to take it.

Sockets: the thing you actually program

You never manipulate IP addresses and ports directly. You ask the operating system for a socket, and it hands you an object with two useful methods: send, and receive.

Socket

A socket is your program's telephone. You ask the OS for one, tell it which port to claim, and then you can send data out and receive data in. All the genuinely hard networking — routing, Wi-Fi radio, error checking, retries — happens beneath it and is not your problem.

The API is essentially four operations: create, bind (claim a port), send, receive. That's most of network programming.

Testing without a phone

Here's a trick that will save you enormous frustration. You now have two unfinished halves. If you build both and it doesn't work, the bug could be anywhere and you'll be debugging Android and Windows simultaneously.

So don't. Write a throwaway script that pretends to be a phone.

Milestone 2 · the server listens

Goal: a Python script on your PC drives the virtual pad.

  1. Add a socket to your Chapter 4 server: bind port 27500, loop on receive.
  2. For now, read two numbers out of each packet and feed them to SetAxisValue. The real packet format is Chapter 8 — anything works today.
  3. Run the Python script below in another window.
  4. joy.cpl — the stick sweeps, driven across a network socket.
# fake_phone.py — pretends to be the client
import socket, struct, math, time

sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
angle = 0.0

while True:
    x = int(math.cos(angle) * 32000)
    y = int(math.sin(angle) * 32000)
    packet = struct.pack('<hh', x, y)          # two 16-bit numbers
    sock.sendto(packet, ('127.0.0.1', 27500))
    angle += 0.05
    time.sleep(0.008)                          # 125 times a second
127.0.0.1

That address means "this same machine." It lets the script and the server talk without any network involved at all — useful because it removes Wi-Fi from the equation while you're getting the basics right.

When you later test from a real phone, swap it for your PC's actual address (find it with ipconfig).

Firewalls, and why your first real phone test will failexpect this

The moment you move from 127.0.0.1 to a real phone over Wi-Fi, Windows Firewall will very likely block it silently. This catches everyone.

Windows usually shows a permission prompt the first time your program binds a port — click Allow, and make sure Private networks is ticked. If you dismissed it, find your app under Windows Defender Firewall → "Allow an app through firewall."

Two related classics worth knowing before they cost you an evening:

  • Your PC is on a different network than you think. Ethernet on one subnet, phone on a guest Wi-Fi. They can't reach each other.
  • AP isolation. Some routers — especially guest networks — deliberately prevent devices from talking to each other at all. Nothing you write can work around it; you'd need to change the network.
What you now know
  1. An IP address identifies a machine. Home addresses (192.168.…) are private to your network — both devices must be on the same Wi-Fi.
  2. A port identifies which program on that machine gets the data. Building, then flat number.
  3. A socket is your program's phone line: create, bind, send, receive.
  4. The server binds a fixed known port so it can be found. The client doesn't need to.
  5. Fake the phone with a short script so the server is debuggable on its own — and keep that script forever as a regression test.
  6. 127.0.0.1 means "this machine," which removes the network from the picture while you get things working.
