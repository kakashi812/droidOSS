Chapter Nine deeper

Glass becomes a joystick

A flat sheet of glass has no edges, no springs, and no way to tell your thumb where it is. Making it feel like a controller anyway is mostly a handful of small decisions — each of which is easy to get subtly wrong.

Builds on Ch 8: the 20-byte packet the phone must produce. And Ch 6: fixed-rate sending at 125 Hz.

The one thing glass can't do

A physical stick springs back to centre, stops at its edges, and pushes back against your thumb. Glass does none of that. Every bit of "feel" has to be manufactured in software, and if you skip a step the result is a controller people describe as "floaty" or "unresponsive" without being able to say why.

Fortunately the fixes are small. There are four of them.

Fix 1 — Track fingers by identity, not position

Android reports touches through a stream of events. The naive approach — put a button widget on screen and listen for taps — falls apart immediately, because playing a game means holding a stick with one thumb while tapping buttons with the other.

Pointer

Android calls each finger currently on the screen a pointer. Multiple pointers exist at once, and each gets an ID that stays constant for as long as that finger stays down.

It also has an index — its slot in the current event's list. The index shifts around as other fingers lift. The ID is stable; the index is not. Confusing them is the single most common bug in touch controls.

So: one full-screen view, and a map from pointer ID to whichever control that finger grabbed when it landed.

override fun onTouchEvent(e: MotionEvent): Boolean {
    when (e.actionMasked) {
        MotionEvent.ACTION_DOWN,
        MotionEvent.ACTION_POINTER_DOWN -> {
            val i  = e.actionIndex
            val id = e.getPointerId(i)
            hitTest(e.getX(i), e.getY(i))?.let { control ->
                pointerToControl[id] = control        // decided once, kept
                control.onDown(e.getX(i), e.getY(i))
            }
        }
        MotionEvent.ACTION_MOVE -> {
            // one MOVE event carries EVERY active finger
            for (i in 0 until e.pointerCount) {
                val id = e.getPointerId(i)
                pointerToControl[id]?.onMove(e.getX(i), e.getY(i))
            }
        }
        MotionEvent.ACTION_UP,
        MotionEvent.ACTION_POINTER_UP,
        MotionEvent.ACTION_CANCEL -> {
            val id = e.getPointerId(e.actionIndex)
            pointerToControl.remove(id)?.onUp()
        }
    }
    return true
}
Four ways this goes wrong
  • Using index instead of ID. The bug above. Feels random, which makes it miserable to debug.
  • Reading only pointer 0 on MOVE. A single ACTION_MOVE contains updates for all fingers. Loop over pointerCount or your second thumb simply stops working.
  • Ignoring ACTION_CANCEL. Pulling down the notification shade cancels the gesture. Miss it and the control stays stuck down — in-game, your character runs into a wall until you notice.
  • Re-testing which control is under the finger on every move. Decide at DOWN and keep it. Otherwise sliding off a button hands your finger to whatever's underneath.

Fix 2 — The stick's geometry

A stick is simpler than it looks: remember where the thumb landed, and measure the vector from there to where it is now.

fun onMove(x: Float, y: Float) {
    var dx = x - originX
    var dy = y - originY
    val dist = hypot(dx, dy)

    if (dist > radius) {          // thumb went past the edge — clamp it
        dx *= radius / dist
        dy *= radius / dist
    }

    axisX = ((dx / radius) * 32767f).toInt().coerceIn(-32768, 32767).toShort()
    axisY = ((-dy / radius) * 32767f).toInt().coerceIn(-32768, 32767).toShort()
    //        ↑ the Y inversion
}
Deadzone

A small region around the centre where input is reported as zero. Real sticks need one because they don't mechanically return to perfect centre. Touchscreens need one because your thumb trembles and because "I meant to hold still" should mean still.

Do it radially — measure the distance from centre, subtract the deadzone from that distance, and rescale what's left across the full range. Doing it per-axis is the classic mistake, and it makes diagonal movement feel like it's snapping to the compass points.

Fix 3 — Let the stick come to the thumb

The single biggest improvement in feel, and it's about ten lines.

Fix 4 — Send on a clock, not on touches

Chapter 6 established that sending at a fixed rate beats sending on change. Here's how that actually looks: the touch handler only updates variables. A separate thread does all the sending, on a timer.

// A dedicated thread. Never the main/UI thread.
val buf    = ByteArray(20)                                    // allocated ONCE
val packet = DatagramPacket(buf, buf.size, serverAddr, serverPort)
val intervalNs = 8_000_000L                                   // 125 Hz

var next = System.nanoTime()
while (running) {
    serializeCurrentState(buf)      // overwrite the same array in place
    socket.send(packet)

    next += intervalNs
    val sleep = next - System.nanoTime()
    if (sleep > 0) LockSupport.parkNanos(sleep)
    else next = System.nanoTime()   // we fell behind — resync, don't spiral
}
Two details doing real work

One buffer, reused forever. The loop allocates no memory at all, so it never triggers garbage collection — the one thing in a managed runtime that could cause a genuinely visible stutter. This is exactly the argument from Chapter 8 for choosing fixed binary over JSON, arriving in practice.

The resync line. Without it, one delayed iteration makes the loop try to "catch up" by firing several packets back to back, then the next one, and so on. You get a burst that's worse than the original hiccup. Falling behind should mean give up on the lost time, not repay it.

Milestone 3 · the project becomes real

Goal: play an actual game with your phone.

  1. One screen. A text field for the server IP, hardcoded default.
  2. Four rectangles for Ⓐ Ⓑ Ⓧ Ⓨ, one circle for a stick. No art, no polish.
  3. Pointer-ID tracking, the stick maths above, the send loop above.
  4. Launch a platformer and play it.

Do not polish the UI before you reach this point. Everything you learn about how it should look, you learn from playing — not from designing.

Stopping Android from sabotaging youessential settings

Three platform behaviours will otherwise quietly ruin the experience:

Wi-Fi power saving. Android parks the radio between packets to save battery, adding tens of milliseconds unpredictably. Hold a high-performance lock while the app is in front:

val lock = wifiManager.createWifiLock(
    WifiManager.WIFI_MODE_FULL_HIGH_PERF, "pad")
lock.acquire()          // and release() in onPause

Screen timeout. Set FLAG_KEEP_SCREEN_ON, or the screen dims mid-game because you haven't "touched" it in a way Android counts.

Edge gestures. Use immersive mode to hide the system bars. Otherwise a thumb near the top edge pulls down the notification shade during a boss fight — which also fires ACTION_CANCEL, so handle that properly too.

Adding gyro aiminggoing further

DroidJoy advertises accelerometer support, but the gyroscope is the more useful sensor. An accelerometer measures acceleration including gravity, so it's noisy and can't tell rotation from movement. Gyro measures rotation directly.

Use TYPE_GAME_ROTATION_VECTOR rather than the raw gyro. It's already fused with other sensors and drift-corrected, and unlike TYPE_ROTATION_VECTOR it excludes magnetometer data — which goes haywire near speakers, magnets, and metal desks.

sensorManager.registerListener(listener,
    sensorManager.getDefaultSensor(Sensor.TYPE_GAME_ROTATION_VECTOR),
    SensorManager.SENSOR_DELAY_GAME)

Map angular offset from a recorded origin onto the right stick. Then add a recentre button — not optional. Drift accumulates, players change posture, and without a recentre they'll be aiming at the floor within a minute.

What you now know
  1. Track fingers by pointer ID, never by index — indices shift when other fingers lift.
  2. One ACTION_MOVE event carries every active finger; loop over all of them.
  3. Handle ACTION_CANCEL, or controls stick down when the notification shade opens.
  4. Decide which control a finger owns at touch-down, and keep it.
  5. Stick maths: vector from origin, clamp to the radius, scale to ±32767, and invert Y.
  6. Deadzone radially, not per-axis, or diagonals snap to the compass points.
  7. A floating origin is the biggest feel improvement for the least code.
  8. Send from a fixed-rate thread using one reused buffer, and resync rather than catch up when late.
