Chapter Four building

Making a fake gamepad appear

Forty lines of code, and a controller that doesn't exist starts moving on your screen. No phone, no networking — just the hard part, done first.

Builds on Ch 3: ViGEmBus is the driver, ViGEmClient is the library, your code stays in user mode. You'll need: a Windows PC.

Why this comes first

Every instinct says to start with the phone — it's the visible part, the fun part. Resist it.

The driver layer is the only piece of this project that could turn out to be impossible on your machine. Maybe the driver won't install. Maybe your Windows version fights it. Everything else — sockets, touch handling, UI — is ordinary programming that definitely works. So we attack the one genuine unknown on day one, while the project has cost you nothing.

What a controller's state looks like in code

Chapter 1 said a controller reports a complete snapshot. Windows defines that snapshot as three groups of numbers:

The last two groups are easy — they're just numbers. Trigger squeezed halfway? That's 128. Stick pushed fully left? That's -32768. Nothing surprising.

The buttons group is the one worth slowing down for, because Windows stores all sixteen of them inside a single number. Let's build up to how.

Step 1: think of a row of light switches

Imagine sixteen light switches in a row, one per button. Each is either up or down. To describe the whole controller's buttons, you just write down which switches are flipped.

Step 2: write "on" as 1 and "off" as 0

Replace each switch with a digit. Now the whole thing is a string of sixteen 1s and 0s — and that string is just a number, written in binary.

Hexadecimal (hex)

Sixteen binary digits in a row is exhausting to read and easy to miscount. So programmers group them in fours and write each group as a single symbol — 0000→0, 0011→3, 1111→F. That's hex.

The 0x prefix just means "the following is hex, not an ordinary decimal number." So 0x3000 is nothing exotic — it's the same sixteen switches, written in four characters instead of sixteen.

Step 3: each button has its own value

If only one switch is on, the whole row still forms a number — that button's own value. Which is why every button has a fixed number attached to it:

ButtonOnly that switch on, in binaryIn hexAs a plain number
Ⓐ0001 0000 0000 00000x10004096
Ⓑ0010 0000 0000 00000x20008192
Ⓧ0100 0000 0000 00000x400016384
Ⓨ1000 0000 0000 00000x800032768

And holding Ⓐ and Ⓑ together means flipping both switches on — which is the same as adding their values:

Ⓐ alone   0x1000     (4096)
Ⓑ alone   0x2000     (8192)
Ⓐ and Ⓑ   0x3000    (12288)   ← both switches on

In code you'd write that with | rather than +. It's called bitwise OR, and it means "flip on every switch that's on in either one." For separate buttons the result is identical to adding — | is just the operator that says what you actually mean.

Bitmask

The general name for this trick: packing many yes/no answers into one number, one binary digit each. You'll meet it everywhere in low-level code — file permissions, network flags, graphics options — so it's worth the ten minutes.

Two operators go with it: | to switch things on, and & to ask whether something is on (if (buttons & 0x1000) reads as "is Ⓐ held?").

You do not have to do any of this by hand

That was worth understanding because you'll see raw hex values in documentation and in other people's code, and Chapter 8 sends this exact number across the network.

But the C# library gives you named buttons and handles the bit-fiddling itself:

pad.SetButtonState(Xbox360Button.A, true);
pad.SetButtonState(Xbox360Button.B, true);

No hex, no |, no counting bits. If Step 3 didn't fully land, it genuinely won't block you — come back to it when you meet 0x1000 in the wild.

Picking a language — and why it matters less than you think

We'll use C#. Here's the honest reasoning, because you should be suspicious of a recommendation with no argument behind it.

You might expect that a project about input lag demands the fastest possible language. Chapter 6 covers this properly, but the short version: the code you write in this layer runs in microseconds, inside a system where the network and the game's own frame timing cost tens of milliseconds. Your language choice is thousands of times too small to matter.

So choose for the thing that does matter — how quickly you can get something working:

OptionThe case forThe case against
C# ✅ Nefarius.ViGEm.Client is maintained by the driver's own author, so the tricky bridge into kernel-land is already solved. Sockets, tray icons, and settings are all one-liners. Nothing that matters here.
C ViGEmClient is a C library, so this is the most direct possible route. And you already know C. The driver call isn't the problem — everything around it is. A Windows GUI, tray icon, and config storage in raw C is a lot of Win32 plumbing unrelated to what you're learning.
Rust Would teach you the most. Learning Rust's ownership model and this architecture at once is two hard things simultaneously.
Not a permanent decision

Once the whole thing works and you know exactly what the server does, porting it to C is a weekend. That's a much better time to do it — you'll be translating a known design rather than discovering one. Starting in C means spending most of your time on scaffolding instead of on the ideas.

Building it

Milestone 1 · your first virtual controller

Goal: a controller that isn't there, visibly moving.

  1. Install the driver. Get the ViGEmBus installer from its releases page and run it. It'll ask for admin rights — that's the driver install we discussed. Reboot if prompted.
  2. New console project. dotnet new console -o PadServer
  3. Add the library. dotnet add package Nefarius.ViGEm.Client
  4. Write the code below, then dotnet run.
  5. Watch it. Press Win+R, type joy.cpl, hit enter. Your phantom controller is listed. Open its properties and the stick is moving in circles.
using Nefarius.ViGEm.Client;
using Nefarius.ViGEm.Client.Targets;
using Nefarius.ViGEm.Client.Targets.Xbox360;

// Connect to the ViGEmBus driver.
using var client = new ViGEmClient();

// Create a virtual Xbox 360 pad and "plug it in".
var pad = client.CreateXbox360Controller();
pad.Connect();          // ← Windows enumerates a controller RIGHT HERE

Console.WriteLine("Virtual pad connected. Open joy.cpl to watch. Ctrl+C to quit.");

double angle = 0;
while (true)
{
    // Sweep the left stick around a circle.
    short x = (short)(Math.Cos(angle) * 32000);
    short y = (short)(Math.Sin(angle) * 32000);

    pad.SetAxisValue(Xbox360Axis.LeftThumbX, x);
    pad.SetAxisValue(Xbox360Axis.LeftThumbY, y);

    // Tap the A button once per full rotation.
    pad.SetButtonState(Xbox360Button.A, Math.Sin(angle) > 0.95);

    pad.SubmitReport();   // ← push the snapshot to the driver

    angle += 0.05;
    Thread.Sleep(16);     // roughly 60 times a second
}

That's the entire hard part of the project. Everything remaining is getting good numbers into those SetAxisValue calls from a phone instead of from Math.Cos.

Try this before moving on

Leave it running and open Steam → Settings → Controller. Steam lists an Xbox 360 controller. Launch any game with controller support and watch your character walk in circles.

Nothing in Steam or that game knows anything about your program. That's the whole idea from Chapter 3, working.

The one design decision to make now

Chapter 3 flagged that ViGEmBus is unmaintained. The fix costs about five minutes today and saves a rewrite later: never let the rest of your program call ViGEm directly. Put a small interface in between.

public interface IVirtualPad : IDisposable
{
    void Plug();
    void Unplug();
    void Submit(PadState state);
    event Action<byte, byte> OnFeedback;   // rumble — Chapter 11
}
The same thing in C, if you'd ratheralternative

ViGEmClient is a C library, so this is a direct translation — six functions in total:

PVIGEM_CLIENT client = vigem_alloc();
vigem_connect(client);

PVIGEM_TARGET pad = vigem_target_x360_alloc();
vigem_target_add(client, pad);        // enumerated now

XUSB_REPORT report = { 0 };
report.wButtons = XUSB_GAMEPAD_A;
report.sThumbLX = 16384;
vigem_target_x360_update(client, pad, report);

vigem_target_remove(client, pad);
vigem_target_free(pad);
vigem_disconnect(client);
vigem_free(client);

vigem_target_add is the moment the controller appears; vigem_target_remove unplugs it. Four players means four PVIGEM_TARGET handles — that's Chapter 11.

If something didn't worktroubleshooting
  • Exception on new ViGEmClient() — the driver isn't installed or the machine hasn't rebooted since. This is by far the most common failure.
  • Installer refuses — you need administrator rights. On a managed work laptop, driver installation may be blocked by policy entirely; you'd need a personal machine.
  • Nothing in joy.cpl — check the program is still running. The controller is unplugged the instant your process exits, by design.
  • Stick moves but games ignore it — likely Steam Input applying its own layer on top. Turn it off for that specific game while testing; it's a classic evening-waster.
What you now know
  1. Attack the riskiest layer first — it's the only one that might be impossible.
  2. A controller's whole state is 12 bytes: a 16-bit button bitmask, two trigger bytes, four stick values.
  3. Your language barely affects latency, so pick for development speed. C# plus Nefarius.ViGEm.Client.
  4. pad.Connect() is the moment Windows enumerates a controller; SubmitReport() pushes each new snapshot.
  5. You have a working virtual gamepad. The rest of the book replaces Math.Cos with a phone.
  6. Hide ViGEm behind a four-method interface now, so the unmaintained driver stays swappable.
