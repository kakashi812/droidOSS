Chapter Three gentle

The trick: a controller made of software

Chapter 2 found the opening. This chapter walks through it — and explains why Steam believes what it's told.

Builds on Ch 2: games ask XInput; XInput asks a kernel-mode driver; nobody can ask "is this real?"

A driver with nothing behind it

Here's the idea in one line: write a driver that reports a controller, without there being a controller.

Everything above it in the stack behaves exactly as it always has. XInput asks the driver for state and gets state. The game asks XInput and gets an answer. Nobody involved is doing anything unusual. The only oddity is that at the very bottom, where you'd expect a USB cable and some plastic, there's a program taking instructions over Wi-Fi.

Virtual bus driver

A bus is the connection system devices plug into — USB is a bus. A virtual bus driver creates a bus that exists only in software, and then reports devices attached to it.

Windows enumerates those devices exactly like physical ones: they appear in Device Manager, they get device IDs, they can be "unplugged." From the operating system's point of view they aren't fake at all — the kernel's definition of "a device exists" is simply "a driver said so."

Why Steam believes it

This is worth being precise about, because people assume something cleverer is going on.

Steam is not being fooled by some hack aimed at Steam. There's no Steam integration to write, no Steam API to call, nothing about Steam in your code at all. Steam asks Windows what controllers are attached, Windows says "an Xbox 360 controller," and Steam reports that faithfully.

Why this matters practically

Because the deception happens below everything, you get all of this without writing a line of code for any of it: correct Ⓐ/Ⓑ/Ⓧ/Ⓨ button prompts in games, Steam Big Picture navigation, Steam Input remapping, controller-based launchers, analog movement, and per-game controller configs.

You didn't add support for those features. They were always going to work the moment a controller existed.

The one you don't have to write

At this point the obvious worry: "I have to write a Windows kernel driver?"

No. Someone already did, and released it free. It's called ViGEmBus.

ViGEmBus can emulate an Xbox 360 controller or a PlayStation DualShock 4. We want the Xbox 360, because that's what XInput speaks and what games assume by default.

One thing to know now, not later

ViGEmBus was retired in November 2023. Its repository is archived and read-only, after a trademark dispute with an unrelated company of a similar name.

It still works. The final signed release installs and runs fine on current Windows 10 and 11, and widely used tools still depend on it. It isn't broken — it's just unmaintained, with nobody obliged to fix it if a future Windows update breaks something.

We'll handle this properly in Chapter 4 by wrapping it behind a small interface, so it can be swapped later without disturbing anything else. Mentioning it here so it isn't a surprise.

The road not taken

Before moving on, let's kill the tempting shortcut properly, because you'll be tempted again around Chapter 4 when driver installation gets fiddly.

The shortcut is to skip the driver and just simulate keyboard and mouse input, which Windows lets any ordinary program do through a function called SendInput. Phone says Ⓐ, server presses Space.

Someone actually shipped the shortcut — here's how it wentcase study

VirtualGamePad is a well-made open-source project in exactly this space. It deliberately refuses to ship a driver, because it wants to be a portable download with no installer — a genuinely appealing goal, and it's how they get onto F-Droid.

The consequence: its reliable, default mode is keyboard/mouse mapping. Its actual gamepad mode is listed on its own roadmap under ongoing experiments as "implemented but experiencing crashes."

This isn't a criticism of them — it's a clean demonstration that the trade-off is real and unavoidable. You cannot have both a driverless install and a dependable virtual controller. Pick one deliberately. We're picking the controller. Full analysis in Chapter 12.

What you now know
  1. The technique is a virtual bus driver — software that reports devices on a bus that doesn't physically exist.
  2. Windows enumerates those devices like real hardware, because the kernel's test for "a device exists" is simply that a driver said so.
  3. Steam isn't being tricked. It asks Windows and reports the answer honestly — the single false statement is made far below it.
  4. You get analog input, correct button prompts, Steam Input, and universal game support for free, as consequences rather than features.
  5. ViGEmBus is the driver (install once); ViGEmClient is the library your normal program calls. You don't write kernel code.
  6. ViGEmBus is retired but functional — we'll isolate it behind an interface so it's replaceable.
  7. Simulating keystrokes is the tempting shortcut, and it's a dead end. Real projects have proven that.
