Chapter Eleven advanced

Rumble, multiplayer, and not breaking

Three things that separate a working demo from something another person can actually use. The third one matters most, and almost nobody builds it first.

Builds on Ch 4 (the pad interface), Ch 8 (message types), Ch 9 (the send loop).

Part one — data flowing backwards

Everything so far has moved in one direction: thumb → phone → PC → game. Rumble reverses it. The game decides you should feel an explosion, and that has to travel all the way back to a motor in your hand.

// Server — ViGEm tells you when the game wants vibration
pad.FeedbackReceived += (sender, e) => {
    // e.LargeMotor and e.SmallMotor are each 0..255
    SendRumblePacket(clientEndpoint, e.LargeMotor, e.SmallMotor);
};
// Phone — one motor, so blend the two
val amplitude = (large * 0.7f + small * 0.3f).toInt().coerceIn(0, 255)
if (amplitude > 0) {
    vibrator.vibrate(VibrationEffect.createOneShot(60L, amplitude))
} else {
    vibrator.cancel()
}
Rate-limit this or regret it

Games update rumble continuously — sometimes every single frame. Forwarding each update as its own vibration call hammers the motor, drains the battery fast, and produces an undifferentiated mush rather than distinct feedback.

Throttle to about 20 Hz, and only send when the value has meaningfully changed. Also check vibrator.hasAmplitudeControl() — on phones without it, any non-zero amplitude means full strength, so you'll want a threshold instead of a blend.

Part two — four players

This one is nearly free, because we planned for it in Chapter 8 with the pad byte.

Trust the sender, not the packet

The packet contains a pad byte, but don't obey it blindly — a buggy or malicious client could claim to be pad 0 and hijack player one. Assign the slot server-side on HELLO, keyed by sender address, and use the byte only as a cross-check.

General rule: the server decides, the client requests.

Part three — the bug that ruins everything

This is the most important section in the chapter, and it's about eight lines of code.

Consider what happens when a phone stops sending. The battery dies. A call comes in. Someone walks out of Wi-Fi range. The app crashes.

Why this is nearly free for us

Because of Chapter 6's decision to send at a fixed rate, silence is unambiguous. A packet is due every 8 ms, so two seconds of nothing can only mean the client is gone.

Had we chosen event-driven sending, silence would be indistinguishable from "the player isn't touching anything," and we'd have needed a separate heartbeat message to tell them apart. Three chapters later, that decision is still paying.

// Run this on a timer, once a second or so
foreach (var client in clients.ToList())
{
    if (now - client.LastPacketTime > TimeSpan.FromSeconds(2))
    {
        client.Pad.Submit(PadState.Zero);   // stop moving FIRST
        client.Pad.Unplug();                // then remove the controller
        clients.Remove(client);
    }
}

Note the order: zero the state before unplugging. Some games latch the last state they saw when a controller disappears, so unplugging while a stick is held can leave the input stuck anyway.

The rest of the sharp edges

Android will suspend your app, and two seconds is too longrobustness

The timeout above is a safety net for crashes and dead batteries. But the common case — the player glances at a notification — is far more frequent and shouldn't take two seconds to resolve.

Handle it explicitly on the phone side. In onPause, send a neutral state and then a BYE:

override fun onPause() {
    super.onPause()
    currentState.reset()          // all buttons up, sticks centred
    sendImmediately()             // don't wait for the next tick
    sendBye()
    sendThread.pause()
    wifiLock.release()
}

Relying on the server's timeout for this is technically correct and practically awful. Two seconds of a stuck stick is very noticeable.

Steam Input will waste one of your eveningsgotcha

Steam has its own controller-remapping layer that sits on top of whatever the OS reports. When it's active, your virtual pad's input passes through Steam's configuration before reaching the game.

The symptoms are maddening: a press registers twice, or maps to something you never sent, or works in one game and not another. You will naturally assume your packet code is wrong and go hunting there.

Before debugging anything input-related, turn Steam Input off for the test game. Right-click the game → Properties → Controller → "Disable Steam Input." If the problem vanishes, it was never yours.

Windows timers are coarser than you'd expectgotcha

Windows' default timer granularity is about 15.6 ms — worse than our entire 8 ms send interval. Thread.Sleep(8) may well sleep for 16.

On the server this mostly doesn't matter, because you're blocked on socket.Receive(), which wakes on data arrival rather than on a timer. But anywhere you write a timed loop, be aware of it.

timeBeginPeriod(1) raises the resolution system-wide. Use it sparingly and reverse it when you're done — it increases power consumption across the whole machine, not just your process.

This app is a battery furnaceux

Screen at full brightness, a Wi-Fi lock preventing the radio from sleeping, a 125 Hz loop, and a vibration motor. Expect heavy drain and tell users so up front — it's much better than them discovering it.

You can drop the send rate when nothing is changing, but keep a floor of about 30 Hz so the timeout logic never misfires during a quiet moment. A stuck-input bug caused by your own battery optimisation would be a particularly annoying way to learn this lesson.

Milestone 4 · someone else can use it

The bar: hand your phone to a friend and say nothing.

  • Discovery works, or the manual fallback is obvious
  • Disconnection is handled — no stuck inputs, ever
  • Missing ViGEmBus produces a helpful message and a link, not a crash
  • A tray icon showing who's connected
  • Rumble, if you've got it
What you now know
  1. Rumble travels backwards through the entire stack, arriving via ViGEm's feedback callback.
  2. A real pad's two motors collapse into the phone's one — weight the large motor higher.
  3. Rate-limit rumble to ~20 Hz or you'll get battery drain and mush.
  4. Four pads maximum — that's XInput's own limit, not your design's.
  5. One UDP socket serves all clients; map sender address → pad slot. The server assigns slots, the client requests them.
  6. The most important robustness rule: on silence, zero the state and unplug — and zero before unplugging.
  7. Fixed-rate sending is what makes silence detectable at all. That Chapter 6 decision is still earning.
  8. Handle onPause explicitly rather than waiting for the timeout.
  9. Disable Steam Input before debugging input problems.
